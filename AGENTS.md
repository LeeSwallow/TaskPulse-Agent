# AGENTS.md

## 목적

이 문서는 이 저장소에서 agent와 개발자가 공통으로 따라야 할 기본 개발 플로우를 정의한다. 기본 원칙은 `GitHub Flow + TDD` 이며, 구현 완료의 기준은 "코드 작성"이 아니라 "테스트 통과 후 머지 가능한 상태"다.

## 브랜치 전략

이 저장소는 `GitHub Flow` 를 사용한다.

- `main` 은 항상 배포 가능하거나 최소한 안정적인 상태를 유지한다.
- 모든 작업은 `main` 에서 새 브랜치를 생성해 진행한다.
- 기능, 버그 수정, 문서 작업 모두 short-lived branch 로 작업한다.
- `main` 에 직접 작업하거나 직접 push 하지 않는다.
- 작업이 끝나면 Pull Request 기준으로 리뷰 가능한 상태를 만든 뒤 머지한다.

권장 브랜치 이름 예시:

- `feature/scheduled-event-api`
- `feature/calendar-ui`
- `fix/scheduler-timezone-bug`
- `docs/prd-refinement`

## 기본 개발 플로우

모든 기능 작업은 아래 순서를 기본값으로 따른다.

1. `main` 기준 최신 상태에서 작업 브랜치를 만든다.
2. 구현할 기능을 명확히 정의한다.
3. 요구사항과 수용 기준을 먼저 확인하거나 보완한다.
4. 테스트를 먼저 작성한다.
5. 테스트가 실패하는 것을 확인한다.
6. 해당 테스트를 통과시키는 최소 구현을 작성한다.
7. 테스트를 다시 실행해서 통과를 확인한다.
8. 필요한 리팩터링을 수행한다.
9. 전체 관련 테스트를 다시 실행한다.
10. PR 기준으로 머지 가능한 상태인지 점검한다.
11. 검증이 끝난 변경만 `main` 으로 merge 한다.

요약하면 기본 흐름은 아래와 같다.

`branch 생성 -> 특정 기능 정의 -> 테스트 구현 -> 기능 구현 -> 테스트 완료 -> PR -> git merge`

실무적으로는 테스트가 먼저이며, 기능 구현은 테스트를 통과시키기 위한 최소 단위로 진행한다.

## TDD 규칙

- 새 기능, 버그 수정, 회귀 방지는 기본적으로 테스트를 동반해야 한다.
- 테스트 없이 구현만 먼저 하는 방식은 예외 상황이 아니면 허용하지 않는다.
- 테스트는 기능 요구사항과 직접 연결되어야 한다.
- 하나의 테스트는 하나의 행동 또는 규칙을 검증해야 한다.
- 실패하는 테스트를 먼저 만든 뒤 구현한다.
- 테스트 통과 후 중복 제거와 구조 개선을 진행한다.

## 작업 단위 규칙

- 한 번에 하나의 기능 단위로 작업한다.
- 기능 단위마다 요구사항, 구현, 테스트 범위를 명확히 구분한다.
- 관련 없는 수정은 같은 작업에 섞지 않는다.
- 문서 수정만 제외하고, 동작 변화가 있는 변경은 테스트가 있어야 한다.

## 머지 기준

다음 조건을 만족해야만 머지 대상으로 본다.

- 작업 브랜치가 명확한 단일 목적을 가진다.
- 기능 요구사항이 충족된다.
- 해당 기능 테스트가 존재한다.
- 새 테스트와 기존 관련 테스트가 모두 통과한다.
- 실패 원인을 무시한 채 우회하지 않는다.
- 임시 코드, 디버그 코드, 주석 처리된 실험 코드는 제거한다.
- 문서 또는 인터페이스 변경이 있으면 관련 문서를 갱신한다.

## Pull Request 규칙

- PR 은 하나의 목적만 가져야 한다.
- PR 설명에는 배경, 변경 내용, 테스트 결과, 남은 리스크를 적는다.
- 리뷰어가 변경 이유와 검증 방법을 바로 이해할 수 있어야 한다.
- CI 또는 로컬 검증이 끝나지 않은 상태로 merge 하지 않는다.
- 리뷰 중 발견된 문제는 새 커밋으로 수정한다.

## 테스트 실행 원칙

- 변경 범위에 가장 가까운 테스트부터 먼저 실행한다.
- 이후 관련 모듈 전체 테스트를 실행한다.
- 머지 전에는 최소한 변경 영향권 전체를 다시 검증한다.
- 테스트 실행 명령과 결과는 작업 기록에 남길 수 있어야 한다.

## Agent 작업 지침

- agent는 구현 요청을 받으면 먼저 기능 범위와 수용 기준을 확인한다.
- 이후 테스트 전략을 정하고 테스트 파일 또는 테스트 케이스부터 작성한다.
- 테스트가 실패하는 상태를 기준선으로 삼는다.
- 그 다음 최소 구현을 진행한다.
- 테스트 통과를 확인하기 전에는 완료로 간주하지 않는다.
- 테스트가 없는 상태에서 `git merge` 를 전제로 작업을 마무리하지 않는다.

## 예외

아래 경우에는 테스트 작성 방식이 달라질 수 있다.

- 초기 문서화 단계
- 단순 설정 파일 추가
- 외부 시스템 제약으로 자동화 테스트가 아직 불가능한 인프라 작업

이 경우에도 가능한 한 검증 방법을 문서로 남겨야 한다.

## 권장 커밋/리뷰 흐름

1. `main` 에서 작업 브랜치 생성
2. 기능 단위 요구사항 확인
3. 테스트 추가
4. 최소 구현
5. 테스트 통과
6. 리팩터링
7. 재검증
8. PR 생성 및 리뷰
9. merge

## 저장소 원칙

- BE는 `uv` 기반 Python 프로젝트 규칙을 따른다.
- FE는 React 기반 프런트엔드 규칙을 따른다.
- 문서 기준 결정 사항은 `docs/` 에 먼저 반영한다.
- 세부 구현 문서는 각 영역의 `BE/docs/`, `FE/docs/` 에 둔다.
- 구현은 문서, 테스트, 코드 순서의 일관성을 유지한다.

# PRD: TaskPulse Agent

## 1. 문서 목적

이 문서는 `TaskPulse Agent` 의 제품 요구를 정의한다. TaskPulse Agent 는 데스크톱 앱에서 캘린더 형식으로 이벤트를 등록하면, 지정된 시간에 agent 가 허용된 tool/MCP 를 사용해 업무를 수행하고 결과를 알려주는 서비스다.

## 2. 제품 형태

- 제품 형태: 데스크톱 앱
- UI: React
- 앱 셸: Electron
- FE-BE 주 통신: gRPC
- 보조 HTTP: `/health`, `/metrics`, 내부 진단

## 3. 문제 정의

반복적이거나 시점 의존적인 업무는 사람이 직접 시간을 기억하고 수동으로 실행해야 하는 경우가 많다. 기존 자동화 도구는 스케줄링, 에이전트 실행, 결과 확인, 도구 제한이 분리되어 있다.

TaskPulse Agent 는 이를 하나로 묶는다.

- 사용자는 데스크톱 앱의 캘린더 UI 에서 이벤트를 등록한다.
- 각 이벤트에는 실행 시각, instruction, 허용 tool/MCP, 결과 전달 대상을 함께 저장한다.
- 실행 시점이 되면 agent 가 허용된 범위 안에서만 작업한다.
- 결과는 데스크톱 앱과 외부 채널에서 확인할 수 있다.

## 4. 핵심 계층

제품의 기본 사용자 계층은 다음과 같다.

- `User`
- `Workspace`
- `Agent`
- `Event`
- `Execution`

모든 이벤트와 실행 기록은 반드시 특정 `Agent` 에 속해야 하고, 모든 `Agent` 는 특정 `Workspace`, 모든 `Workspace` 는 특정 `User` 에 속해야 한다.

## 5. 제품 비전

사용자가 자연어로 업무를 등록하고, 시스템이 시간 기반으로 안전하게 실행하는 데스크톱 중심의 스케줄형 에이전트 운영 도구를 만든다.

핵심 방향:

- 일정 등록은 캘린더 UX 로 단순해야 한다.
- 실행 범위는 허용된 tool/MCP 로 제한되어야 한다.
- 결과는 운영자가 빠르게 스캔할 수 있게 구조화되어야 한다.
- 초기 버전은 단일 로컬 사용자 데스크톱 앱으로 시작하되, 데이터 모델은 다중 사용자 확장을 고려한다.

## 6. MVP 범위

### 6.1 포함

- Electron 기반 데스크톱 앱 셸
- 캘린더 기반 이벤트 등록/수정/삭제
- Workspace / Agent 선택
- 이벤트별 instruction 저장
- 이벤트별 허용 tool/MCP 목록 선택
- 1회성 일정, 일간 반복, 주간 반복
- FE-BE 간 `gRPC` 통신
- 백엔드 스케줄러가 예정 시각에 작업 실행
- LangGraph 기반 agent execution flow
- 실행 결과 저장
- 결과 조회 UI
- 수동 재실행

### 6.2 제외

- 서버 SaaS 멀티 테넌시
- 복잡한 권한 체계
- 결제/플랜
- 외부 워크플로 엔진 통합
- 고급 반복 규칙(RRULE 전체 지원)
- 장기 메모리/자기개선 에이전트

## 7. 성공 지표

### 7.1 제품 지표

- 등록된 이벤트 수
- 실제 예약 실행 성공률
- 실행 후 결과 조회율
- 수동 재실행 비율

### 7.2 시스템 지표

- 예약 시각 대비 실행 지연 시간
- tool 실행 성공률
- 이벤트 실패율
- 평균 실행 시간
- desktop app start-to-ready 시간

## 8. 핵심 시나리오

1. 사용자가 데스크톱 앱을 연다.
2. 사용자는 workspace 와 agent 를 선택한다.
3. 사용자는 캘린더에서 내일 오전 9시 이벤트를 등록한다.
4. 사용자는 "어제 업계 뉴스를 수집하고 요약해서 Slack 으로 알려줘" 라고 입력한다.
5. 사용자는 허용 tool 로 `web_search`, `send_slack` 를 선택한다.
6. 지정된 시각에 백엔드 스케줄러가 이벤트를 감지한다.
7. agent 는 허용된 tool 만 사용하여 작업을 수행한다.
8. 결과 요약, tool 사용 로그, 실행 상태가 저장된다.
9. 사용자는 데스크톱 앱에서 결과를 확인하거나 실패 시 재실행한다.

## 9. 제약 및 원칙

- 에이전트는 이벤트에 등록된 허용 tool 범위를 넘어서면 안 된다.
- 실행 결과는 재현 가능한 로그를 남겨야 한다.
- 예약 실행은 서버 재시작 후에도 복구 가능해야 한다.
- FE-BE 주 통신은 gRPC 를 기준으로 한다.
- 프런트엔드는 사용자/워크스페이스/에이전트 컨텍스트를 항상 유지해야 한다.

## 10. 기술 방향

- BE: `uv`, `Python`, `gRPC`, `LangChain`, `LangGraph`
- FE: `React`, `Electron`
- 데이터 저장: `PostgreSQL`, `pgvector`, `Redis`
- 통신: `gRPC` as primary protocol
- 스케줄링: 앱 내부 scheduler 우선, 필요 시 외부 job runner 확장

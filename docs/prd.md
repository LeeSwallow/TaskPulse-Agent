# PRD: 등록형 Scheduled Agent Service

## 1. 문서 목적

이 문서는 "캘린더 형식으로 이벤트를 등록하면, 지정된 시간에 agent가 허용된 tool/MCP를 사용해 업무를 수행하고 결과를 알려주는 서비스"의 제품 요구를 정의한다. 현재 단계는 아이디에이션 및 구현 준비 단계이며, 세부 구현보다 문제 정의와 범위 합의에 초점을 둔다.

## 2. 문제 정의

반복적이거나 시점 의존적인 업무는 보통 사람이 직접 시간을 기억하고 수동으로 실행해야 한다. 기존 자동화 도구는 스케줄링은 잘하지만 자연어 기반 업무 정의, 에이전트 실행, 허용 도구 제한, 결과 요약/통지가 분리되어 있다.

이 서비스는 아래를 하나로 묶는다.

- 사용자는 캘린더 UI에서 이벤트를 등록한다.
- 각 이벤트에는 "언제 실행할지", "무엇을 할지", "어떤 tool/MCP를 사용할 수 있는지", "결과를 어디로 알려줄지"를 함께 저장한다.
- 실행 시점이 되면 agent가 해당 범위 안에서만 작업을 수행한다.
- 실행 결과는 요약과 로그 형태로 사용자에게 전달된다.

## 3. 제품 비전

사용자가 자연어로 업무를 등록하고, 시스템이 시간 기반으로 안전하게 실행하는 "스케줄형 에이전트 오케스트레이션 서비스"를 만든다.

핵심 방향은 다음과 같다.

- 일정 등록은 캘린더 UX로 단순해야 한다.
- 실행 범위는 허용된 tool/MCP로 제한되어야 한다.
- 결과는 사람이 빠르게 검토할 수 있게 구조화되어야 한다.
- 초기 버전은 단일 사용자 중심 MVP로 시작한다.

## 4. 대상 사용자

### 4.1 1차 사용자

- 개인 생산성 자동화를 원하는 개발자
- 반복 업무를 위임하고 싶은 PM/운영 담당자
- 특정 시각에 리서치, 요약, 알림 업무를 수행하고 싶은 팀

### 4.2 초기 사용 맥락

- 매일 아침 시장/뉴스 요약
- 주간 회의 전 아젠다 문서 생성
- 특정 시간에 Slack/Notion/Jira 연동 작업 실행
- 반복 점검 결과를 대시보드나 메시지로 보고

## 5. 핵심 가치 제안

- 자연어 기반 작업 등록
- 시간 기반 자동 실행
- 허용 도구 제한으로 안전성 확보
- 실행 결과의 기록, 재실행, 감사 가능성

## 6. MVP 범위

### 6.1 포함

- 캘린더 기반 이벤트 등록/수정/삭제
- 이벤트별 instruction 저장
- 이벤트별 허용 tool/MCP 목록 선택
- 1회성 일정, 일간 반복, 주간 반복
- 백엔드 스케줄러가 예정 시각에 작업 실행
- LangGraph 기반 agent execution flow
- 실행 결과 저장
- 결과 조회 UI
- 수동 재실행

### 6.2 제외

- 멀티 테넌시
- 복잡한 권한 체계
- 결제/플랜
- 외부 워크플로 엔진 통합
- 고급 반복 규칙(RRULE 전체 지원)
- 장기 메모리/자기개선 에이전트

## 7. 성공 지표

### 7.1 제품 지표

- 등록된 이벤트 수
- 실제 예약 실행 성공률
- 실행 후 결과 조회율
- 수동 재실행 비율

### 7.2 시스템 지표

- 예약 시각 대비 실행 지연 시간
- tool 실행 성공률
- 이벤트 실패율
- 평균 실행 시간

## 8. 핵심 시나리오

1. 사용자가 캘린더에서 내일 오전 9시 이벤트를 등록한다.
2. 사용자는 자연어로 "어제 업계 뉴스를 수집하고 요약해서 Slack으로 알려줘"라고 입력한다.
3. 사용자는 허용 tool로 `web_search`, `send_slack`를 선택한다.
4. 지정된 시각에 백엔드 스케줄러가 이벤트를 감지한다.
5. agent는 허용된 tool만 사용하여 작업을 수행한다.
6. 결과 요약, tool 사용 로그, 실행 상태가 저장된다.
7. 사용자는 대시보드에서 결과를 확인하거나 실패 시 재실행한다.

## 9. 제약 및 원칙

- 에이전트는 이벤트에 등록된 허용 tool 범위를 넘어서면 안 된다.
- 실행 결과는 재현 가능한 로그를 남겨야 한다.
- 예약 실행은 서버 재시작 후에도 복구 가능해야 한다.
- 프런트엔드와 백엔드는 독립 실행 가능해야 한다.

## 10. 기술 방향

- BE: `uv`, `FastAPI`, `LangChain`, `LangGraph`
- FE: `React`, 현재는 `Vite` 기반 시작
- 저장소: MVP 단계에서는 파일 또는 SQLite 우선 검토
- 스케줄링: 앱 내부 scheduler 우선, 필요 시 외부 job runner 확장

## 11. 오픈 이슈

- 실제 MCP 연동 방식을 어떻게 표준화할 것인가
- 결과 통지 채널을 Slack 우선으로 할지, 대시보드 우선으로 할지
- 장기 실행 작업에 대한 timeout/retry 정책을 어디까지 둘지
- 사용자 인증을 MVP에서 제외할지 간단 로그인만 둘지
